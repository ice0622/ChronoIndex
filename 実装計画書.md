## 1. 背景と動機

### 1.1 AIエンジニアとしての価値創造

AIエンジニアの需要は急速に高まっており、スカウトが増える一方で、汎用的なスキルセットだけでは差別化が難しい状況になっている。この状況において自分の価値を作るために、「課題を特定してPoCで解決する」というアプローチを選択する。

単に「AIが使える」ではなく、**具体的な業務課題に対してソリューションを実証できる実績**を積み重ねることが、独自のポジションを確立する上で最も実効的な手段である。

### 1.2 解決する課題

YouTube等の動画プラットフォームには、8時間を超える長尺コンテンツが多数存在する（カンファレンス、学会発表、社内勉強会、講義録画など）。しかし、その多くにはチャプター（タイムスタンプ付き目次）が付与されていない。

**現状の問題:**
- 視聴者が特定の発表・セクションを探す際に、全体を再生して手動でシークしなければならない
- 動画の価値が埋もれる（情報のアクセシビリティが低い）
- 手動でチャプターを作成するコストが高く（特に8時間超の動画では数時間かかる）、後回しにされる

**理想の状態:**
```
10:00 開会式
13:00 ○○株式会社 ○○の発表
55:30 質疑応答
1:02:00 ○○株式会社 ○○の発表
...
7:45:00 閉会式・表彰
```
このような目次が自動で生成される。

## 2. プロジェクト概要

### 2.1 プロジェクト名

**ChronoIndex**（クロノインデックス）

> Chrono（時間）× Index（索引）  
> 時間軸に沿った意味のある索引を自動生成するというコンセプト

### 2.2 ミッション

長尺動画の音声を解析し、**YouTubeチャプター形式のタイムスタンプ付き目次**を自動生成する。

### 2.3 価値展開の構造

```
PoC（技術検証）
    ↓
「長尺動画の構造化」というテーマでの価値確立
    ↓
横展開（会議録 / 大学講義 / 社内ナレッジ動画 / 医療カンファレンス等）
```

## 4. 技術方針

### 4.1 基本方針：音声ベースのシンプルなパイプライン

PoCの目的は**技術的な実現可能性を早期に証明すること**であるため、マルチモーダル統合は行わず、音声のみで実装する。後から映像情報を追加できるアーキテクチャにしておく。

```
[音声抽出]
    ↓
[音声認識（ASR）]  → タイムスタンプ付きトランスクリプト
    ↓
[セグメント境界検出]  → 話者変化・沈黙・言語パターン
    ↓
[LLMによるチャプタータイトル生成]  → 各セグメントの要約
    ↓
[YouTube形式への整形・バリデーション]
    ↓
[出力：タイムスタンプ付き目次テキスト]
```

### 4.2 技術スタック（候補）

| レイヤー | 技術候補 | 選定理由 |
|---|---|---|
| 音声抽出 | yt-dlp | YouTube URLから直接音声を取得可能 |
| ASR | Whisper (large-v3) / WhisperX | 日本語精度が高い、タイムスタンプ付き出力 |
| 話者分離 | WhisperX + pyannote-audio | 話者ダイアライゼーション対応 |
| チャプター生成 | GPT-4o / Claude 3.5 Sonnet | 長コンテキスト、日本語要約精度 |
| バリデーション | 独自ロジック（Python） | YouTube仕様への適合チェック |
| 実行環境 | Python + Docker | 再現性の確保 |

### 4.3 セグメント境界の検出ロジック

音声ベースのPoC段階では、以下の3つの手法を組み合わせる。

**① 言語パターンマッチング（ルールベース）**

司会・進行者が発する定型フレーズを境界マーカーとして検出する。

```python
BOUNDARY_PATTERNS = [
    r"それでは.*(セッション|時間|部|発表)を.*開始",
    r"続きまして.*登壇者は",
    r"ありがとうございました",
    r"質疑応答(の時間)?を?始め",
    r"以上で.*終了",
    r"休憩",
    r"開会",
    r"閉会",
]
```

**② 沈黙区間検出（音響ベース）**

3秒以上の無音区間は、発表者の交代や場面転換の可能性が高い。

**③ 話者変化（話者ダイアライゼーション）**

WhisperX + pyannote-audioにより、話者IDが変化する時点を境界候補とする。

### 4.4 YouTube チャプター仕様への適合ルール

出力フォーマットは以下の制約を自動チェックする。

- 最初のタイムスタンプは必ず `00:00` から始まる
- チャプター数は3つ以上
- 各チャプターの長さは10秒以上
- タイムスタンプは昇順
- フォーマット: `MM:SS タイトル` または `H:MM:SS タイトル`

## 9. 参考リサーチの知見（要約）

別ドキュメント「[youtube動画構造化POC計画.md](youtube動画構造化POC計画.md)」にてGENIAC-PRIZE最終審査動画を対象に実施したリサーチから、以下の重要な知見を得ている。

### 汎用化できる知見

1. **言語パターンはドメイン間で部分的に共通**  
   「それでは」「ありがとうございました」「質疑応答」といったフレーズは、日本語の会議・発表系コンテンツに広く共通して存在する。

2. **沈黙 + 話者変化の組み合わせが最も信頼できる境界シグナル**  
   単一のシグナルより複数を組み合わせることで、偽陽性を大幅に減らせる。

3. **固有名詞の補正にコンテキスト知識が不可欠**  
   音声認識が誤変換した固有名詞をLLMが周辺文脈から修正できるかが精度の鍵。

4. **チャプタータイトルは「テーマ + 話者 + 核心」の3要素が望ましい**  
   例：`【製造業】ダイキン工業 × Fairy Devices：熟練者の技をAIで継承する`

5. **先行研究（Chapter-LlamaベースのLLMアプローチ）はF1スコア0.45超を達成**  
   従来手法（0.27程度）を大きく上回る。本PoCの技術的実現可能性を裏付ける。

### 12.2 難所ごとの計測方法の詳細

#### D-1 WER（単語誤り率）の計測

```python
# evaluate_asr.py のコアロジック
from jiwer import wer, cer

def measure_asr_quality(hypothesis: str, reference: str) -> dict:
    """
    hypothesis : Whisperが出力した文字起こし
    reference  : 人手で作成した正解テキスト
    """
    return {
        "WER": wer(reference, hypothesis),   # 単語誤り率（低いほど良い）
        "CER": cer(reference, hypothesis),   # 文字誤り率（日本語はこちらが主指標）
    }

# 目標: CER ≤ 0.10（日本語、クリーン音声）
# 目標: CER ≤ 0.20（ノイズあり、専門用語多め）
```

#### D-3 境界検知 MAE の計測

```python
# evaluate_boundary.py のコアロジック
import numpy as np

def evaluate_boundary_detection(
    predicted_timestamps: list[float],  # 自動生成（秒）
    ground_truth_timestamps: list[float],  # 正解（秒）
    tolerance: float = 30.0  # マッチングの許容ズレ
) -> dict:
    errors = []
    matched_gt = set()
    for pred in predicted_timestamps:
        # 正解の中で最も近いものとマッチング
        dists = [(abs(pred - gt), i) for i, gt in enumerate(ground_truth_timestamps)
                 if i not in matched_gt]
        if dists:
            dist, idx = min(dists)
            if dist <= tolerance:
                errors.append(dist)
                matched_gt.add(idx)

    tp = len(errors)
    fp = len(predicted_timestamps) - tp
    fn = len(ground_truth_timestamps) - tp

    precision = tp / (tp + fp) if (tp + fp) > 0 else 0
    recall    = tp / (tp + fn) if (tp + fn) > 0 else 0
    f1        = 2 * precision * recall / (precision + recall) if (precision + recall) > 0 else 0

    return {
        "MAE_sec":   np.mean(errors) if errors else float("inf"),
        "Precision": precision,
        "Recall":    recall,
        "F1":        f1,
        "TP": tp, "FP": fp, "FN": fn,
    }

# 目標: MAE ≤ 10秒, F1 ≥ 0.75
```

## 13. ハルシネーション対策とその検証フレームワーク

> **想定される質問:** 「LLMがでたらめなタイトルを生成しないようにどう工夫しましたか？」「その工夫が効いていることをどう確認しますか？」

ハルシネーション（LLMが事実にない内容を生成すること）は、チャプタータイトルの信頼性を根本から損なう。本プロジェクトでは「防止」と「検出」の二重構造で対策する。

---

### 13.1 ハルシネーションのパターン分類

本プロジェクトで起こりうるハルシネーションを具体的に分類する。

| タイプ | 具体例 | 影響 |
|---|---|---|
| **H-1 存在しない登壇者名の生成** | トランスクリプトに出てこない企業名・人名がタイトルに含まれる | 視聴者への誤情報、信頼性の喪失 |
| **H-2 発言内容の誇張・捏造** | 「A社がBを達成した」と言っていないのにそう書かれる | 事実誤認の拡散 |
| **H-3 タイムスタンプの論理矛盾** | 実際に話された内容と時刻がずれている | ユーザーが目的の場所に飛べない |
| **H-4 専門用語の意味誤解** | 「PLaMo」を「プラモデル（玩具）」の文脈で解釈してタイトル生成 | ドメイン知識の欠如が露呈 |

---

### 13.2 防止策（Prevent Layer）

#### 対策① グラウンディング強制プロンプト

LLMに対して「トランスクリプトの中にある情報だけを使ってタイトルを生成すること」を明示的に指示する。

```python
GROUNDED_PROMPT = """
あなたはYouTube動画のチャプタータイトルを生成するアシスタントです。
```

#### 対策② チャンクのスコープ制限

LLMには「そのチャプターに対応するトランスクリプトのチャンクのみ」を渡す。動画全体を渡すと、前後の文脈から存在しない関連を推論するリスクが高まる。

```
[誤った設計]  全トランスクリプト → LLM → 全チャプター生成（ハルシネーションリスク大）

[正しい設計]  
境界検出 → チャンク [t0, t1] → LLM → タイトル1
           チャンク [t1, t2] → LLM → タイトル2
           チャンク [t2, t3] → LLM → タイトル3
```

#### 対策③ Temperature を低く設定する


### 13.3 検出策（Detect Layer）

防止策だけでは不十分なため、生成後に自動検証する。

#### 検証① エンティティグラウンディングチェック

生成されたタイトルから固有名詞を抽出し、対応チャンクのトランスクリプトにその名詞が存在するかを確認する。

```python
def check_entity_grounding(title: str, transcript_chunk: str, nlp_model) -> dict:
    """
    title             : LLMが生成したチャプタータイトル
    transcript_chunk  : そのチャプターのトランスクリプト
    """
    title_entities = [ent.text for ent in nlp_model(title).ents]
    chunk_text     = transcript_chunk.lower()

    grounded     = [e for e in title_entities if e.lower() in chunk_text]
    not_grounded = [e for e in title_entities if e.lower() not in chunk_text]

    grounding_score = len(grounded) / len(title_entities) if title_entities else 1.0

    return {
        "grounding_score":  grounding_score,   # 1.0が理想（全固有名詞がchunkに存在）
        "grounded_entities":     grounded,
        "hallucinated_entities": not_grounded, # ここに値があればハルシネーション候補
        "is_safe": grounding_score >= 0.9,
    }

# 目標: grounding_score ≥ 0.90（全チャプターの平均）
```

### 13.4 ハルシネーション計測ダッシュボード指標

実装完了後、以下の指標を評価レポートに掲載する。

| 指標名 | 定義 | 目標値 | 計測タイミング |
|---|---|---|---|
| **Grounding Score** | タイトル内の固有名詞のうちトランスクリプトに存在した割合 | ≥ 0.90 | 全チャプター・自動計測 |
