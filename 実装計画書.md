# ChronoIndex 実装計画書
## 長尺動画の自動構造化 POC

**作成日:** 2026年3月25日  
**ステータス:** 実装前計画（Draft v1.0）

---

## 1. 背景と動機

### 1.1 AIエンジニアとしての価値創造

AIエンジニアの需要は急速に高まっており、スカウトが増える一方で、汎用的なスキルセットだけでは差別化が難しい状況になっている。この状況において自分の価値を作るために、「課題を特定してPoCで解決する」というアプローチを選択する。

単に「AIが使える」ではなく、**具体的な業務課題に対してソリューションを実証できる実績**を積み重ねることが、独自のポジションを確立する上で最も実効的な手段である。

### 1.2 解決する課題

YouTube等の動画プラットフォームには、8時間を超える長尺コンテンツが多数存在する（カンファレンス、学会発表、社内勉強会、講義録画など）。しかし、その多くにはチャプター（タイムスタンプ付き目次）が付与されていない。

**現状の問題:**
- 視聴者が特定の発表・セクションを探す際に、全体を再生して手動でシークしなければならない
- 動画の価値が埋もれる（情報のアクセシビリティが低い）
- 手動でチャプターを作成するコストが高く（特に8時間超の動画では数時間かかる）、後回しにされる

**理想の状態:**
```
10:00 開会式
13:00 ○○株式会社 ○○の発表
55:30 質疑応答
1:02:00 ○○株式会社 ○○の発表
...
7:45:00 閉会式・表彰
```
このような目次が自動で生成される。

## 2. プロジェクト概要

### 2.1 プロジェクト名

**ChronoIndex**（クロノインデックス）

> Chrono（時間）× Index（索引）  
> 時間軸に沿った意味のある索引を自動生成するというコンセプト

### 2.2 ミッション

長尺動画の音声を解析し、**YouTubeチャプター形式のタイムスタンプ付き目次**を自動生成する。

### 2.3 価値展開の構造

```
PoC（技術検証）
    ↓
「長尺動画の構造化」というテーマでの価値確立
    ↓
横展開（会議録 / 大学講義 / 社内ナレッジ動画 / 医療カンファレンス等）
```

## 3. スコープ定義

### 3.1 PoCスコープ（本計画の対象）

| 項目 | 内容 |
|---|---|
| 対象動画 | YouTube上の公開長尺動画（1時間〜8時間程度） |
| 入力 | 動画URL または 音声ファイル（mp4 / mp3 / wav） |
| 出力 | YouTube概要欄に貼り付け可能なタイムスタンプ付きチャプターテキスト |
| モダリティ | **音声ベース**（映像解析はPoC範囲外） |
| 言語 | 日本語（メイン）、英語（対応可能な範囲で） |

### 3.2 PoC対象の具体的ユースケース

技術検証の主対象として以下を想定する。

1. **技術カンファレンス動画**（GENIAC-PRIZE最終審査などを参考にしたリサーチ済みドメイン）
2. **大学講義の録画動画**（90〜120分、章立てが明確）
3. **社内技術勉強会のアーカイブ**（複数発表者、Q&Aあり）

### 3.3 スコープ外（将来展開）

- リアルタイム処理（ライブ配信への対応）
- 映像フレーム解析・スライドOCR
- YouTube APIへの自動投稿
- 多言語対応

## 4. 技術方針

### 4.1 基本方針：音声ベースのシンプルなパイプライン

PoCの目的は**技術的な実現可能性を早期に証明すること**であるため、マルチモーダル統合は行わず、音声のみで実装する。後から映像情報を追加できるアーキテクチャにしておく。

```
[音声抽出]
    ↓
[音声認識（ASR）]  → タイムスタンプ付きトランスクリプト
    ↓
[セグメント境界検出]  → 話者変化・沈黙・言語パターン
    ↓
[LLMによるチャプタータイトル生成]  → 各セグメントの要約
    ↓
[YouTube形式への整形・バリデーション]
    ↓
[出力：タイムスタンプ付き目次テキスト]
```

### 4.2 技術スタック（候補）

| レイヤー | 技術候補 | 選定理由 |
|---|---|---|
| 音声抽出 | yt-dlp | YouTube URLから直接音声を取得可能 |
| ASR | Whisper (large-v3) / WhisperX | 日本語精度が高い、タイムスタンプ付き出力 |
| 話者分離 | WhisperX + pyannote-audio | 話者ダイアライゼーション対応 |
| チャプター生成 | GPT-4o / Claude 3.5 Sonnet | 長コンテキスト、日本語要約精度 |
| バリデーション | 独自ロジック（Python） | YouTube仕様への適合チェック |
| 実行環境 | Python + Docker | 再現性の確保 |

### 4.3 セグメント境界の検出ロジック

音声ベースのPoC段階では、以下の3つの手法を組み合わせる。

**① 言語パターンマッチング（ルールベース）**

司会・進行者が発する定型フレーズを境界マーカーとして検出する。

```python
BOUNDARY_PATTERNS = [
    r"それでは.*(セッション|時間|部|発表)を.*開始",
    r"続きまして.*登壇者は",
    r"ありがとうございました",
    r"質疑応答(の時間)?を?始め",
    r"以上で.*終了",
    r"休憩",
    r"開会",
    r"閉会",
]
```

**② 沈黙区間検出（音響ベース）**

3秒以上の無音区間は、発表者の交代や場面転換の可能性が高い。

**③ 話者変化（話者ダイアライゼーション）**

WhisperX + pyannote-audioにより、話者IDが変化する時点を境界候補とする。

### 4.4 YouTube チャプター仕様への適合ルール

出力フォーマットは以下の制約を自動チェックする。

- 最初のタイムスタンプは必ず `00:00` から始まる
- チャプター数は3つ以上
- 各チャプターの長さは10秒以上
- タイムスタンプは昇順
- フォーマット: `MM:SS タイトル` または `H:MM:SS タイトル`

## 5. 実装フェーズ計画

### Phase 0：環境構築と動作確認（1週間）

| タスク | 内容 |
|---|---|
| 開発環境セットアップ | Python環境、Docker、依存ライブラリのインストール |
| yt-dlp疎通確認 | YouTube URLから音声ファイルのダウンロードが機能することを確認 |
| Whisper動作確認 | 短い音声（5〜10分）での文字起こし精度を目視確認 |
| LLM API疎通確認 | OpenAI / Anthropic APIのレート制限・コストを把握 |

**完了条件:** YouTubeの動画を入力して、タイムスタンプ付きトランスクリプトが得られること

---

### Phase 1：ベースラインパイプラインの構築（2週間）

**目標:** 手動で正解タイムスタンプが付いた動画に対して、自動生成との差分を測定できる状態にする。

| タスク | 内容 |
|---|---|
| エンドツーエンドパイプラインの実装 | yt-dlp → Whisper → LLM → 出力 の一連の流れ |
| 評価用データセットの作成 | 手動タイムスタンプが存在する動画を3〜5本収集 |
| 評価スクリプトの実装 | 自動生成 vs 正解のズレ（秒単位）を計算 |
| 出力フォーマット実装 | YouTube仕様バリデーション付きの出力生成 |

**成功指標:**
- 境界検知の平均誤差: **30秒以内**（ベースライン目標）
- YouTube仕様適合率: **100%**
- 処理時間: 1時間の動画を **10分以内**で処理

---

### Phase 2：精度改善（2〜3週間）

**目標:** ベースラインの誤差を縮め、タイトル品質を向上させる。

| タスク | 内容 |
|---|---|
| 話者ダイアライゼーション統合 | WhisperX + pyannote-audioの組み込み |
| 境界検出ロジックの改善 | 言語パターン + 沈黙 + 話者変化の組み合わせ最適化 |
| プロンプトエンジニアリング | チャプタータイトル生成プロンプトの改善（企業名・テーマを含む形式） |
| 長尺対応 | 3時間以上の動画をウィンドウ分割して処理するロジックの実装 |

**成功指標:**
- 境界検知の平均誤差: **10秒以内**
- 固有名詞（登壇者名・企業名）の認識精度: **90%以上**

---

### Phase 3：検証とドキュメント整備（1週間）

## 7. 横展開ロードマップ

PoCの成功後、以下の順序で適用ドメインを拡大する。

```
[Phase A] 技術カンファレンス
    → スライド構成が明確、登壇者の交代が明確、精度が出やすい

[Phase B] 大学講義動画
    → 単一話者が多い、章立てが明確、教育機関でのニーズあり

[Phase C] 社内会議・社内勉強会動画
    → 最も需要が高い、固有の社内用語への対応が必要

[Phase D] 医療カンファレンス・学会
    → 高い信頼性要求、専門用語辞書の整備が必須
```

各ドメインで共通するコアパイプラインは再利用し、**ドメイン固有の語彙辞書とプロンプトのカスタマイズ**のみで対応する設計にする。

## 8. リスクと対策

| リスク | 影響度 | 対策 |
|---|---|---|
| ASR精度の低下（背景ノイズ、方言） | 高 | Whisper large-v3の採用、前処理（ノイズ除去）の追加 |
| 専門用語の誤変換 | 高 | ドメイン辞書の事前登録、LLMによる後補正 |
| LLMのAPI費用が高騰 | 中 | 処理チャンクの最適化、コスト上限の設定 |
| 長尺動画のメモリ・処理時間 | 中 | ウィンドウ分割処理、バッチ実行への対応 |
| 話者の重なり（Q&A区間での同時発話） | 中 | 話者ダイアライゼーションの導入、ルールベースとの併用 |

## 9. 参考リサーチの知見（要約）

別ドキュメント「[youtube動画構造化POC計画.md](youtube動画構造化POC計画.md)」にてGENIAC-PRIZE最終審査動画を対象に実施したリサーチから、以下の重要な知見を得ている。

### 汎用化できる知見

1. **言語パターンはドメイン間で部分的に共通**  
   「それでは」「ありがとうございました」「質疑応答」といったフレーズは、日本語の会議・発表系コンテンツに広く共通して存在する。

2. **沈黙 + 話者変化の組み合わせが最も信頼できる境界シグナル**  
   単一のシグナルより複数を組み合わせることで、偽陽性を大幅に減らせる。

3. **固有名詞の補正にコンテキスト知識が不可欠**  
   音声認識が誤変換した固有名詞をLLMが周辺文脈から修正できるかが精度の鍵。

4. **チャプタータイトルは「テーマ + 話者 + 核心」の3要素が望ましい**  
   例：`【製造業】ダイキン工業 × Fairy Devices：熟練者の技をAIで継承する`

5. **先行研究（Chapter-LlamaベースのLLMアプローチ）はF1スコア0.45超を達成**  
   従来手法（0.27程度）を大きく上回る。本PoCの技術的実現可能性を裏付ける。
